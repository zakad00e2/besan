# Neon Booking Integration Design

## Goal

Persist appointment requests submitted through `/book-call` in Neon Postgres and display them in the dashboard bookings table on every device. Protect all dashboard customer and appointment data with Neon Auth and restrict access to the configured administrator email.

## Scope

This change includes:

- A new Neon project and Neon Auth provisioned on its default branch.
- A server-only Neon Postgres connection.
- Database schema and migration SQL for customers and appointments.
- Server-side booking creation and dashboard booking reads.
- Authentication UI and dashboard route protection.
- Server-side administrator authorization.
- Loading, validation, duplicate-slot, and failure states.
- Automated domain, server-boundary, authentication, and UI tests.
- An `.env.example` file without credentials.

This change does not include workshop-modal persistence, customer profile migration, WhatsApp delivery, reminder delivery, multiple administrators, role management UI, or production SMTP setup.

## Architecture

Use `@neondatabase/serverless` over HTTP from TanStack Start server functions. The database connection module is server-only and reads `DATABASE_URL`; no database credential is imported into browser bundles.

Use Neon Auth through `@neondatabase/neon-js` and `@neondatabase/auth-ui`. The app exposes a sign-in flow and protects the dashboard in the UI, while every server function that returns customer or appointment data independently validates the current session and checks the signed-in email against `ADMIN_EMAIL`. UI protection is not treated as authorization.

The public booking form calls a narrowly scoped server function. It accepts only booking-form fields, validates and normalizes them, then performs the customer upsert and appointment insert as one non-interactive transaction. It never accepts status, reminder state, administrator identity, or arbitrary database fields from the browser.

## Database Schema

### `customers`

| Column       | Type          | Rules                              |
| ------------ | ------------- | ---------------------------------- |
| `id`         | `uuid`        | Primary key, generated by Postgres |
| `full_name`  | `text`        | Required, trimmed                  |
| `mobile`     | `text`        | Required, normalized for matching  |
| `created_at` | `timestamptz` | Required, defaults to `now()`      |
| `updated_at` | `timestamptz` | Required, defaults to `now()`      |

`mobile` has a unique constraint so repeat bookings reuse one customer record. A later submission updates the stored display name and `updated_at` for that mobile number.

### `appointments`

| Column             | Type          | Rules                                                    |
| ------------------ | ------------- | -------------------------------------------------------- |
| `id`               | `uuid`        | Primary key, generated by Postgres                       |
| `customer_id`      | `uuid`        | Required foreign key to `customers.id`                   |
| `appointment_type` | `text`        | Required and constrained to the five `/book-call` values |
| `appointment_date` | `date`        | Required                                                 |
| `appointment_time` | `time`        | Required                                                 |
| `notes`            | `text`        | Required, empty string allowed, bounded by validation    |
| `status`           | `text`        | Required, defaults to `pending`                          |
| `reminder_status`  | `text`        | Required, defaults to `not-scheduled`                    |
| `created_at`       | `timestamptz` | Required, defaults to `now()`                            |
| `updated_at`       | `timestamptz` | Required, defaults to `now()`                            |

The database stores the selected local atelier date and time as `date` and `time`, rather than inventing a UTC offset that the form does not collect. A partial unique index on `(appointment_date, appointment_time)` for non-cancelled appointments prevents concurrent double-booking while allowing a cancelled slot to be booked again.

## Booking Submission Flow

1. The customer selects an appointment type, available date, and time and enters their name, mobile number, and optional notes.
2. The browser submits those values to a TanStack Start server function.
3. Shared Zod validation checks required fields, allowed appointment types, valid selectable weekdays, allowed times for that weekday, field lengths, and normalized mobile format.
4. The server transaction inserts or updates the customer by normalized mobile number and inserts the appointment with `pending` and `not-scheduled` states.
5. On success, the form displays a confirmed receipt message and prevents accidental repeat submission from the same rendered form state.
6. A duplicate slot returns a specific availability error and keeps the customer's entered details in the form.
7. Other database or network failures show a generic retryable error and are logged server-side without exposing credentials or raw SQL errors.

Client-side validation is retained for responsiveness, but server-side validation is authoritative.

## Authentication and Authorization

Neon Auth provides the sign-in and session flow. The only authorized administrator email for this release is supplied through `ADMIN_EMAIL` and initially configured as `z409483831@gmail.com` in the deployment environment.

Dashboard routes redirect signed-out visitors to the sign-in route. A signed-in account whose normalized email does not match `ADMIN_EMAIL` receives an access-denied screen and no booking data. The server-side booking list function repeats the session and email check before querying Postgres, preventing direct function calls from bypassing the UI.

The public booking submission does not require authentication. It exposes insert-only business behavior through validation and fixed SQL; it does not expose general database reads or updates.

Production launch requires configuring trusted production domains, a production email provider, email verification policy, and disabling localhost access in Neon Auth. Development may use Neon's default development email behavior.

## Dashboard Data Flow

The bookings route loads live rows from Neon after authorization. The server query joins appointments with customers and returns a purpose-built booking view model containing:

- appointment ID;
- customer name and mobile;
- appointment type;
- selected date and time;
- notes;
- appointment status;
- reminder status;
- submission timestamp.

The bookings table displays these fields on desktop. The mobile layout includes the same information in a compact stacked presentation. Existing search and filters operate on the loaded live records. Empty, loading, access-denied, and load-failure states are explicit.

The type filter uses the five appointment types offered by `/book-call`, rather than the prototype's broad `design` and `workshop` categories. Search covers customer name, mobile, appointment type, and notes. Date and status filters continue to work against the live fields.

The current `New appointment` and `Edit appointment` controls only mutate the prototype's in-memory store, so they are removed from the database-backed bookings route in this release. Keeping them would falsely imply that a change was saved. The WhatsApp action remains because it is derived from the stored customer mobile number and does not mutate booking data. Database-backed create, status-change, reschedule, and cancellation controls are separate follow-up work.

The dashboard's existing in-memory demo store remains available to unrelated overview, customer, and availability screens in this release. The bookings route is the first database-backed dashboard surface. Database-backed customer profiles and overview metrics are separate follow-up work to avoid silently mixing incomplete cross-screen behavior into this feature.

## Environment Configuration

`.env.example` documents:

```dotenv
DATABASE_URL=postgresql://USER:PASSWORD@HOST/neondb?sslmode=require
VITE_NEON_AUTH_URL=https://YOUR-AUTH-HOST/neondb/auth
ADMIN_EMAIL=z409483831@gmail.com
```

The real `.env` remains ignored by Git. `DATABASE_URL` and `ADMIN_EMAIL` are server-only. `VITE_NEON_AUTH_URL` is intentionally public because the browser auth client needs the Auth endpoint.

## Migration and Project Setup

Create a Neon project named `besan2` in the connected organization and provision Neon Auth on its default branch. Apply idempotent migration SQL through Neon's safe migration workflow, review the branch diff, and only then apply the migration to the main branch.

The migration enables `pgcrypto` if needed, creates the two tables, constraints, and indexes, and leaves existing Neon Auth schemas untouched. A migration file is committed to the repository so the schema is reproducible outside the connector session.

## Testing

Follow test-driven development:

- Domain tests for normalization, validation, allowed days and times, and database-row mapping.
- Server repository tests for parameterized queries, customer reuse, appointment insertion, duplicate-slot translation, and sanitized failure behavior.
- Authorization tests for signed-out, wrong-email, and authorized-admin cases.
- Form interaction tests for successful submission, duplicate slot, validation errors, retryable failure, and repeat-submit prevention.
- Dashboard tests proving that live booking rows, notes, status, empty state, and load errors render correctly.
- Regression tests for the retained booking search, filters, responsive rendering, and WhatsApp action.

Final verification includes focused tests, the full test suite, ESLint, a production build, and manual checks of `/book-call`, `/auth/sign-in`, and `/dashboard/bookings` at desktop and mobile widths.

## Success Criteria

- A valid `/book-call` submission creates one customer and one pending appointment in Neon.
- A repeat mobile number reuses the customer record.
- Concurrent attempts for the same active slot cannot both succeed.
- A successful booking appears in `/dashboard/bookings` on another device after an authorized sign-in.
- Signed-out and non-admin users cannot read customer or appointment data through either the UI or direct server calls.
- All submitted fields, including notes and submission time, are available in the bookings view.
- No database credential or raw database error is exposed to the browser or committed to Git.
